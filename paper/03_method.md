## Structure (6 subsections)

### 3.1 Three-arm tool surface (~0.30 page)

Define each arm by what tools are exposed to the agent. The harness applies the restriction at the agent-process level — the same model binary runs all three arms.

| Arm | Allowed surface | Claude Code (claude_code) | Codex CLI (codex_cli) |
|---|---|---|---|
| `tool_rich` | Full IDE / native surface: file primitives + shell + structured patch | `build_tools_flags("tool_rich", …)` returns `[]` (no restriction flag) — [runner.py:211](../swebench/runner.py#L211) | `config.toml [features]` with `shell_tool = true`, `apply_patch_freeform = true` — [runner.py:965-968](../swebench/runner.py#L965-L968) |
| `bash_only` | Shell only; all file-handling primitives disallowed | `--tools Bash --disallowedTools <all-others>` — [runner.py:222-226](../swebench/runner.py#L222-L226) | `shell_tool = true`, `apply_patch_freeform = false`; structured `apply_patch` cannot be disabled and is soft-discouraged via a prompt prefix ([_BASH_ONLY_DIRECTIVE](../swebench/runner.py#L801)) |
| `code_only` | Single MCP tool `mcp__codebox__execute_code` (Python + Bash inside a persistent REPL kernel) | `--mcp-config … --strict-mcp-config --tools mcp__codebox__execute_code,mcp__codebox__list_tools --disallowedTools <all-builtins>` — [runner.py:213-220](../swebench/runner.py#L213-L220) | `shell_tool = false`, `apply_patch_freeform = false`, codebox MCP registered; structured `apply_patch` soft-discouraged via [_CODE_ONLY_DIRECTIVE](../swebench/runner.py#L788) |

Two implementation asymmetries the prose must surface explicitly:

1. **Codex `apply_patch` cannot be hard-disabled.** Codex exposes no feature flag for its structured `apply_patch` tool, so on Codex the `bash_only` and `code_only` arms enforce restriction via a prompt-prefix directive. This is soft enforcement; model compliance is not guaranteed. Disclose this — a reviewer comparing surface restrictions across agents will catch it otherwise. See [_apply_arm_directive](../swebench/runner.py#L808).
2. **`code_only` requires a persistent REPL.** The exec-server stack ([exec_server/](../exec_server/): `exec-server.js` + `codebox.py` + `python_kernel.py`) maintains a Python interpreter keyed by cwd across `execute_code` invocations. Without persistence the round-trip savings of "one script per task" collapse — an ablation we run in §5 (forward-reference). Persistence is gated by `ONLYCODES_PERSISTENT_KERNEL=1`.

**Naming.** SWE-bench legacy code uses `baseline` / `onlycode` for the same semantics as artifact's `tool_rich` / `code_only`. The paper uses the artifact names throughout.

**Mini-SWE-agent disclosure.** The `bash_only` arm matches mini-SWE-agent's *tool surface* (bash only) but does not reuse its loop, prompt template, or scaffold code \citep{repo:miniswagent} — we apply the restriction to the same Claude Code / Codex binaries the other two arms use. This is the correct framing: we are isolating the *tool-surface* variable, not comparing scaffold implementations.

### 3.2 Harness design (~0.30 page)

Per-instance setup (SWE-bench only; artifact tasks are self-contained):

- **Repo clone + venv setup.** Clone the instance's repo at `base_commit` and build a per-instance Python venv against it. The repo and the venv are each given their own pristine OverlayFS *lower*, so a per-arm upperdir captures all writes and is discarded between arms — no re-clone, no re-`pip install -e .`. The venv-side overlay is mounted back at the original venv creation path because console-script shebangs bake that absolute path at install time. See [cache.py](../swebench/cache.py) and [run.py](../swebench/run.py).
- **Git-history strip.** [`strip_git_history`](../swebench/harness.py) collapses the repo to a single orphan commit at the current worktree's tree, deletes all refs / packed-refs / reflogs, repacks into local objects, removes the alternates file, and runs `git gc --prune=now`. The agent cannot recover the upstream reference fix via `git log` / `git show` / reflog. Fixed author and committer dates make the orphan SHA deterministic for a given tree (idempotent re-runs).
- **Overlay refresh between arms.** `git reset` cannot un-create files an earlier arm added (fuse-overlayfs copy-up + EEXIST semantics), so [`_refresh_overlay`](../swebench/run.py) unmounts → deletes upperdir+workdir → recreates → remounts → re-strips history. The agent's view of the merged mount path is stable; the underlying state is fresh.

Per-arm setup:

- **Subprocess isolation.** Each invocation creates a fresh temp config directory and points the agent at it via `CLAUDE_CONFIG_DIR` ([ClaudeRunner.invoke](../swebench/runner.py)) or `CODEX_HOME` ([CodexRunner._make_isolated_config](../swebench/runner.py)). The directory holds only the credentials and minimal config needed for the run; no session-persistence files, no cross-run state. Claude is invoked with `--dangerously-skip-permissions --no-session-persistence`; Codex with `--dangerously-bypass-approvals-and-sandbox`.
- **Wall-time cap.** Each invocation is wrapped in `subprocess.Popen(..., start_new_session=True)` + `proc.wait(timeout=…)`; a `TimeoutExpired` triggers `os.killpg(..., SIGKILL)` on the agent's process group (so subagents and spawned shells die with the parent) and the run is marked FAIL with a synthetic `{"type":"system","subtype":"wall_timeout"}` JSONL record. Default budget is the harness-level `wall_timeout_seconds` argument; specific value pinned in §4 via `\result{wall_budget_seconds}`.
- **Per-task prompt-cache isolation (optional).** The `--cache-isolation` flag mints a fresh 16-hex nonce per (instance, arm, run_idx) — [`generate_isolation_nonce`](../swebench/runner.py#L41) — and injects it into the outbound tool surface as a stub MCP server. For Codex (issue #294) the nonce is serialized into the `tools[]` array, breaking OpenAI's prefix-cache hash. For Claude Code (issue #296) the symmetric mechanism registers an `iso_nonce` MCP server. **The mechanism is necessary-but-insufficient — Claude's stub races startup so the nonce does not always land before the first API call.** Failure mode and resulting reporting methodology are owned by §3.5.

### 3.3 Evaluation integrity (~0.4 page)

SWE-bench's canonical evaluation protocol applies the gold `test_patch` only after the agent terminates: at run time the agent observes the repository at `base_commit`, and the tests used for grading are not present in the worktree \citep{jimenez2024swebench}.

Our harness implements deferred application explicitly. [`apply_test_patch`](../swebench/harness.py) runs only after the agent's process exits. Once the patch is applied, a `pytest --collect-only` gate runs against the patched tree to confirm that the held-out tests are collectible — a zero-item collection scores FAIL rather than letting `run_tests` silently report success against a non-existent suite. Because our agents — unlike the bash-only patch-submission scaffold used in the original SWE-bench evaluation — can write arbitrary files anywhere in the worktree, the canonical "apply the patch" step is preceded by two cleanup operations:

1. Every file that appears as a target in `test_patch` is `git checkout HEAD -- <path>`'d to discard any agent edits.
2. Every path the patch lists as a "new file" is removed from the worktree, so an agent-created file cannot collide with a held-out test file and block the apply.

`git apply` is then invoked against the resulting tree. If it still fails — almost always because an agent-created file could not safely be removed in step (2) — the instance is recorded FAIL with reason `"post-agent git apply failed; agent edits conflicted with the held-out test patch"`, rather than scored against a tree we cannot trust. This is strictly more conservative than the canonical protocol, which assumes a clean apply.

**Audit.** We additionally ran an audit cross-checking, for every reported SWE-bench instance, whether the agent attempted to read or modify any file targeted by the gold `test_patch`, and — for instances where it did — whether removing that read/modify event from the trace would have changed the final verdict. The propagation rate from "agent touched a test-target file" to "verdict would differ" is reported in §5 (`\result{audit_total}`, `\result{audit_propagated}`). This separates the *possibility* of an integrity leak from its *consequence* on the headline numbers.

### 3.4 The Capability Overlap framing (~0.25 page)

Zhang et al. (Apr 2026; cite via `\cite{zhang2026tooltax}` once approved into `references.bib`) formalize the tool-use tax inequality: a tool provides net benefit iff the task-specific capability gain it delivers exceeds the per-call cost of carrying its definition in the prompt. We apply this framing to coding-agent IDE primitives — the original work targets general math/QA tool use, not coding tools.

Under Capability Overlap, the §6 redundancy table reads as follows: most Claude Code IDE primitives are bash-subsets in *capability* (Read/Grep/Glob/Write reduce to `cat`/`grep`/`find`/heredoc); `Edit` is the one primitive with non-overlapping capability (atomic byte-precise replace with lint, which is awkward to express in bash). The per-call tax (system-message bytes for the tool definitions) is task-invariant; the *capability gain* depends on regime. On computation tasks where there is no codebase structure to exploit, gain ≈ 0 and the tax dominates → `code_only` wins. On modification + multi-file tasks, IDE primitives' exploration efficiency offsets the tax → `tool_rich` wins.

The sign-flip in §5 is the empirical signature of the Capability Overlap inequality crossing zero between regimes. Section 3.4 establishes the frame; quantitative claims (tax magnitude, primitive-by-primitive overlap) live in §6.

### 3.5 Cache isolation and cost reporting (~0.3 page)

Target length **5–7 sentences in compiled prose**. The bracketed text below is the planning guide; condense aggressively when drafting.

**[Fact — No vendor offers a cache-isolation knob.]** Across the four surfaces this paper touches (Anthropic Messages API, Claude Code CLI, OpenAI Responses API, OpenAI Codex CLI), no documented parameter, flag, environment variable, or namespace knob forces a cache miss or scopes cached tokens to a single caller. The only documented mechanism is implicit: change the cached content so the cache key changes. OpenAI's `prompt_cache_key` is a routing/partition hint, not a disable switch; Anthropic's `cache_control` controls breakpoints, not bypass; Claude Code's `--no-session-persistence` controls local session state, not the upstream API cache. The supporting vendor documentation is staged in [02_related_work.md](02_related_work.md) under "Vendor documentation (cited in §3.5)" with matching BibTeX in the staging block — the planned cite-points for the compiled prose are `\citep{docs:anthropic2026promptcaching, docs:anthropic2026contextediting, docs:claudecode2026cli, docs:claudecode2026settings, docs:openai2026promptcaching, docs:openai2026convstate, repo:openai2026codex}` (do **not** auto-insert into `references.bib`; await approval per repo policy). We therefore cannot guarantee per-task cache isolation in the reported numbers and instead address the resulting non-stationarity in the cost-reporting methodology below.

**[Consequence — cache-adjusted cost reporting.]** Cost in §5 is reported as a single cache-adjusted column, defined from the underlying token formula plus a per-arm median-floor adjustment:

- **Token cost formula.** `input_tokens × full_input_rate + output_tokens × output_rate` — every token billed at non-cached rates. Reproducible from raw JSONL `usage` records; immune to provider cache-state variation. This is the underlying input to the cache-adjusted column, not a parallel reporting line.
- **Median-floor adjustment.** For each arm, let `M_arm` be the *median* across that arm's instances of first-turn `cache_read_input_tokens` — empirically the floor representing "system prompt + tool definitions are warm in cache". For any instance whose first-turn `cache_read` falls **below** `M_arm` (cache-cold by accident of run order), we bump it up to `M_arm` for accounting — i.e., we credit that instance as if it had hit the cache for the arm's shared prefix. Within-task multi-turn cache reads (turns ≥ 2) and any instance already at or above `M_arm` are unchanged. This normalizes the system-prompt warm/cold lottery without claiming we achieved isolation.

**[Why the adjustment is bounded at the system prompt.]** The adjustment deliberately stops at the arm-median, which corresponds to the size of the shared system prompt + tool definitions; we do not claim it controls for cache sharing **beyond** that prefix. Such post-system-prompt cross-task sharing is, however, vanishingly unlikely. Both providers' prompt caches key on a prefix hash that is invalidated by any byte difference in the cached span. Past the system prompt, the conversation diverges per task: the first user message is the task statement (a fresh string per instance), the assistant's response is task-conditional, and every subsequent tool call carries task-specific arguments. For a cache hit to extend beyond the system prompt, two instances would need byte-identical task statements, byte-identical model responses, and byte-identical tool-call payloads in identical order — an event of negligible probability across 93 artifact + 100 SWE-bench instances. Empirically, per-instance first-turn `cache_read` clusters tightly around the system-prompt token count for each arm, with no bimodal tail consistent with cross-task content reuse.

We deliberately do *not* report raw observed (API-billed) cost: it is sensitive to run order and to ambient cache state we cannot control, and reporting numbers we know are confounded would mislead readers. Cache-adjusted is the single reported cost column.

### 3.6 The artifact-suite contract (~0.3 page; non-negotiable)

The artifact suite is a new benchmark; §3.6 is the methodology layer that defends it the way §3.3 defends our SWE-bench protocol. It exists to support the §1 benchmark-contribution claim and to make the §5 computation-regime numbers reproducible from the released code. **Per-category counts and what each category probes are §4 material; this section describes the contract, not the inventory.**

**Task contract.** Every artifact instance is a four-part bundle on disk: `task.yaml` (problem statement + execution-budget fields + pointers to the other three parts), `workspace/` (the public files copied into the agent's scratch dir), `grader/hidden.py` (the grader exposing a single `grade(scratch_dir) → GradeResult` entry point), and `grader/reference_output.*` (a reference artifact used to verify the grader itself). Schema and instance-ID convention (`<category>__<slug>`) are specified in [docs/SCHEMA_ARTIFACT.md](../docs/SCHEMA_ARTIFACT.md); loading and field validation live in [artifact_loader.py](../swebench/artifact_loader.py). Execution-budget fields (`max_code_runs`, `max_wall_seconds`) are *declared* in `task.yaml` but **enforcement is off** for v1 — disclosed here so a reader of the YAML does not conclude otherwise.

**Grader contract.** Each task's `grader/hidden.py:grade(scratch_dir)` returns a `GradeResult` with a score ∈ [0, 1] and a `passed: bool`; the harness invokes it via [artifact_grade.py](../swebench/artifact_grade.py). Three rules constrain what `grade()` may do, and are checked by [test_verify_graders.py](../tests/test_verify_graders.py): **(i) deterministic** — re-grading the same `scratch_dir` returns the same score; **(ii) offline** — no network calls, no API keys, no clock-dependent oracles; **(iii) seeded-random only** — any stochasticity is gated by a fixed seed declared in the grader. `grade()` must not write to `scratch_dir` (would contaminate later re-grading). The reference-output round-trip — `grade(workspace/ + reference_output/)` must return `score == 1.0` — is the grader's own self-test; it is executed by `test_verify_graders.py` and CI on every push.

**Subprocess isolation.** Graders run in a fresh subprocess via [_artifact_grade_runner.py](../swebench/_artifact_grade_runner.py), with results serialized as JSON on stdout. A grader-side exception cannot kill the harness; stdout pollution from grader code is forbidden by the same contract. This is the artifact-side analogue of the SWE-bench harness's `--collect-only` pre-flight gate.

**No-leak invariant (pre-flight, not post-hoc).** Before the agent runs, [`materialize()`](../swebench/artifact_materialize.py) copies `workspace/` into the scratch dir and then scans the resulting tree for any path matching `grader/hidden.py` or `reference_output.*`. A match raises `MaterializationError` and the run is aborted *before* the agent starts; no run can complete with the answer key in its working dir. This is a pre-flight check by design — catching a leak post-hoc would already have biased the grade.

**Citing back.** §4 (Experimental Setup) reports per-category counts and a one-line description of what each category probes from `paper/data/artifact_categories.csv`. The artifact-suite paragraph in §6 Discussion notes that the four contract rules (deterministic / offline / seeded-random / no-leak) generalize to other coding-agent benchmarks — i.e., what a follow-up author would need to add a category or task. (Per-category pass/cost breakdowns were cut from §5 in the 2026-05-28 restructure.)
